# アダプティブ・プログレッション設計（レベル・XP・3ドメイン・プレースメント）

- 日付: 2026-07-06
- ステータス: 承認済み（ブレインストーミングでの合意を文書化）
- 関連: [システム設計](2026-07-05-learn-english-system-design.md) §5（方法論10原則）、`docs/research/2026-07-05-continuation-research-phase3.md`（ゲーミフィケーション・継続性）

## 1. 目的

練習の難易度が利用者の実力に合わせて**なだらかに**上がっていく仕組みを作る。具体的には:

1. ロールプレイほかのメニューを **日常会話 / 一般ビジネス / IT実務** の3ドメインで満遍なく回す
2. **レベル（無制限に上がる整数）** が難易度のつまみを駆動し、お題・時間圧・準備支援がレベルに応じて変化する
3. **XP（減らない累積値）** で進捗を可視化し、レベルアップの材料にする
4. **プレースメント（初期レベル見極め）** で開始点を測定し、月次の再測定で進歩を定点観測する

## 2. 研究制約（不変条件）

phase3 リサーチの検証済み結論に従う。本設計のすべての要素はこの枠内に収める:

- **情報的フィードバックのみ**。喪失感を煽る演出（ストリーク切れ・減点・降格ペナルティ表現・他者比較・ランキング）は実装しない
- XP は**減らない**。レベルは自動では**下がらない**（降格は「調整の提案」としてのみ提示し、承認制。文言は中立: 「基礎を固め直すこともできます」）
- 昇格・降格の判定根拠は利用者に**開示**する（ブラックボックス化しない）

## 3. レベルモデル

### 3.1 レベルとステージ

- `level`: 1 から始まる整数。**上限なし**
- `stage(level) = min(6, ceil(level / 10))`: 難易度の粗い帯域。**Lv1〜60 が難易度レンジ**（stage 1〜6 ≒ CEFR A2前半〜B2）、**Lv61以降は難易度据え置きのおまけ**（表示に「難易度は最大」を添える）
- 体感が変わる切り替え（準備支援・ヒント言語・モデルトーク・お題帯域）は**ステージ境界でのみ**起きる。ステージ内は連続的なつまみ（秒数）だけが微増する

### 3.2 難易度つまみ（stage → パラメータ表）

| stage | 4/3/2 初回秒（レンジ） | 準備チャンク数 | ヒント言語 | モデルトーク | お題・シナリオ帯域 |
|---|---|---|---|---|---|
| 1 (Lv1-10) | 90〜105 | 8 | 日本語併記 | 自動表示 | level タグ 1 を含むもの |
| 2 (Lv11-20) | 105〜120 | 7 | 日本語併記 | 自動表示 | 同 2 |
| 3 (Lv21-30) | 120〜135 | 6 | 日本語併記 | 自動表示 | 同 3 |
| 4 (Lv31-40) | 135〜150 | 5 | 英語のみ | 自動表示 | 同 4 |
| 5 (Lv41-50) | 150〜165 | 4 | 英語のみ | ボタンで任意表示 | 同 5 |
| 6 (Lv51-60) | 165〜180 | 4 | 英語のみ | なし | 同 6 |

- **4/3/2 のラウンド秒**はレベルで線形補間する: `first(level) = round5(90 + (min(level,60) - 1) * 1.5)`、ラウンド構成は `[first, first*0.75, first*0.5]`（round5 = 5秒単位に丸め）。ミニ版は先頭2ラウンド
- 現行固定値（120/90/60）は Lv21 相当。**プレースメント未実施時のデフォルトは Lv13**（stage 2、現行よりやや易しい＝既存ユーザーの「難しすぎた」フィードバックを反映）
- 表の数値はすべて `app/server/progression.ts` の定数として一元定義し、後から調整可能とする

## 4. XP

### 4.1 獲得（すべて加算のみ）

| 行為 | XP |
|---|---|
| メニューブロック完了（クイックドリル含む） | ブロックの設計分数と同値（例: 音読6分 = 6 XP） |
| 暗記例文300 の1枚評価 | good=2 / soso=1 / bad=1（努力に対して付与。bad でも減らない） |
| プレースメント/月次測定の完了 | 10 XP |

### 4.2 レベルアップ必要XPとカーブ

- `need(level) = 15 + 5 * stage(level)`（Lv61以降は一定 45）
- 序盤（stage1: 20 XP/レベル）はクイックドリル2日で1レベル上がる感覚。Lv60 到達に必要な総XPは約1,950（1日15XP換算で約4ヶ月半）
- **ステージ内のレベルアップは XP 到達で自動**（変化は秒数の微増のみで事故リスクがないため）
- 数値は定数として一元定義・調整可能

## 5. 昇格・降格（ステージ境界のみ提案＋承認）

### 5.1 ステージ昇格

Lv10→11 など境界をまたぐレベルアップは自動では起きない。以下を満たすとホームに**昇格提案カード**を表示:

- 境界レベルの必要XPに到達している、かつ
- 直近14日間に練習日が5日以上、かつ
- 直近のブロック完了率（開始したブロックのうち完了した割合、直近20ブロック）が 70% 以上

カードには根拠（上記シグナルの実値）を表示し、ワンタップ承認で昇格。却下したら XP はそのまま累積し、7日後に再提案。

### 5.2 ステージ降格（調整の提案）

直近7日間のブロック完了率が 40% 未満、または 4/3/2 の中断が直近5回中3回以上 → 「1ステージ戻して基礎を固め直すこともできます」と提案（中立表現・承認制）。承認するとレベルは現ステージ最下端の1つ下（例: Lv23 → Lv20）へ。**自動降格はしない**。

### 5.3 手動変更

設定からレベルを任意の値に常時変更可能（プレースメントより利用者の自己判断を優先する）。

## 6. プレースメント（初期レベル見極め・月次再測定）

### 6.1 セッション構成（約10分・専用メニュー）

1. **自己紹介**（1分・録音）: 仕事と最近の関心を話す
2. **状況説明**（1.5分・録音）: 提示された日常/仕事の状況（テキスト）を英語で説明する
3. **意見**（1分・録音）: 賛否が分かれる短い問いに意見を述べる

### 6.2 評価

- 各タスクの録音をローカル STT → Claude が **CEFR 記述子ベースの6段階ルーブリック**（stage 1〜6 に対応、プロンプトに明文で埋め込む）で評価
- 客観指標を併用: タスクごとの発話語数、発話密度（語数÷録音秒数）
- 出力: 推定 stage（1〜6）、日本語の根拠2〜3行、開始レベル `=(stage-1)*10+3`（ステージ中央やや下）
- 利用者が確定操作をして初めて反映（「この結果で始める」/「自分で選ぶ」）

### 6.3 再測定

同一タスク構成で月1回の再測定導線をホームに出す（前回から30日経過時）。結果は履歴として保存し、レベルには**自動反映しない**（昇格は §5 の通常経路。再測定は定点観測が目的）。M5 の月次アセスメントはこの履歴を土台にする。

## 7. コンテンツの3ドメイン化

### 7.1 frontmatter 拡張（topics / scenarios 共通）

```yaml
domain: daily | business | it   # 省略時: it（既存コンテンツの実態に合わせる）
level: [1, 6]                   # 適合 stage 範囲。省略時: [1, 6]（全ステージ）
```

- 省略時デフォルトにより**既存コンテンツは無変更でも動く**（後方互換）
- 既存の topics 22本・scenarios 8本には実態に合わせて `domain` と `level` を付与する

### 7.2 追加コンテンツ

- シナリオ: **日常会話4本**（例: 旅行先のトラブル、レストラン、病院、近所の雑談）と**一般ビジネス4本**（例: 日程交渉、進捗報告、苦情対応、面接を受ける側）を新規作成（完全オリジナル・話し言葉）
- 既存IT系8本と合わせて 16本・3ドメイン体制

### 7.3 選択ロジック

現ステージに適合するプール（`level` 範囲に stage を含む）から、**ドメインをラウンドロビン**（前回選んだドメインの次を優先）しつつ、ドメイン内は既存の LRU。プールが空のドメインはスキップ。ロールプレイ・4/3/2のお題・音読ウォームアップすべて同じセレクタを通す。

## 8. データと API

### 8.1 SQLite（既存 data/learn-english.db に追加）

```sql
CREATE TABLE IF NOT EXISTS user_progress (
  id INTEGER PRIMARY KEY CHECK (id = 1),   -- 1行のみ
  level INTEGER NOT NULL,
  xp INTEGER NOT NULL,                     -- 累積（減らない）
  updated_at TEXT NOT NULL
);
CREATE TABLE IF NOT EXISTS xp_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts TEXT NOT NULL, kind TEXT NOT NULL, amount INTEGER NOT NULL, meta TEXT
);
CREATE TABLE IF NOT EXISTS level_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts TEXT NOT NULL, kind TEXT NOT NULL,    -- propose-up / accept-up / decline-up / propose-down / accept-down / manual-set
  from_level INTEGER NOT NULL, to_level INTEGER NOT NULL, rationale TEXT
);
CREATE TABLE IF NOT EXISTS placement_results (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ts TEXT NOT NULL, stage INTEGER NOT NULL, start_level INTEGER NOT NULL,
  rationale TEXT NOT NULL, metrics TEXT NOT NULL   -- JSON: 語数・密度等
);
```

### 8.2 API（RouteDeps に必須フィールドとして追加）

- `GET /api/progress/summary` → `{ level, xp, xpIntoLevel, xpToNext, stage, difficultyMaxed, proposal: null | { kind: "up"|"down", toLevel, rationale } }`
- `POST /api/progress/xp` `{ kind, amount, meta? }` → 付与後の summary（クライアントがブロック完了時等に送る。amount はサーバ側で kind に応じて上限検証）
- `POST /api/progress/level` `{ action: "accept" | "decline" | "set", level? }` → summary
- `GET /api/placement/tasks` → タスク定義3件
- `POST /api/placement/submit` `{ tasks: [{ taskId, transcript, durationSec, wordCount }] }` → `{ stage, startLevel, rationale }`（STT はクライアントが既存 /api/stt を使用）
- `POST /api/placement/confirm` `{ accept: boolean, level? }` → summary
- メニュー系 API（/api/menu/today, /api/menu/quick）は `user_progress` を読み、レベル由来のパラメータでメニューを構築する（HTTP 契約は params の値が変わるのみ）

## 9. UI

- **サイドバー**: 練習記録カードを拡張し「Lv 23」＋次レベルまでのゲージ（xpIntoLevel/need）を表示。Lv61+ は「難易度は最大」を小さく添える
- **ホーム**: 昇格/降格提案カード（根拠つき・承認/却下ボタン・情報トーン）。プレースメント未実施なら「10分でレベル測定」導線カード、実施済みで30日経過なら「月次measurement」導線
- **プレースメント画面**: 3タスクを順に録音 → 結果（stage・根拠・開始レベル）→ 確定
- **レベル手動変更**: サイドバーのレベル表示クリックで変更ダイアログを開く（専用の設定画面は作らない）
- i18n: 追加 UI 文言は `i18n.ts` の辞書に EN/JA 両方で追加（デフォルト英語）

## 10. 実装フェーズ

| フェーズ | 内容 | 依存 |
|---|---|---|
| **A** | frontmatter 拡張（domain/level・後方互換）、日常4本＋一般ビジネス4本のシナリオ追加、ドメインラウンドロビン選択 | なし |
| **B** | progression.ts（レベル・XP・ステージ・つまみ計算）、SQLite テーブル、progress API、menu.ts のレベル駆動化、サイドバーゲージ、昇格/降格提案カード | A |
| **C** | プレースメント（タスク・評価・確定・月次導線） | B |

各フェーズを個別の実装計画（writing-plans）＋サブエージェント駆動で実施。フェーズごとにマージ可能な完結した状態にする。

## 11. テスト方針

- progression.ts: stage/first秒/need の境界値（Lv1, 10, 11, 60, 61, 100）、昇格条件・降格条件のシグナル判定（:memory: SQLite + 固定日付）
- 選択ロジック: ドメインラウンドロビン＋LRU＋帯域フィルタの組み合わせ、プール空ドメインのスキップ、タグなしコンテンツの後方互換
- API: 契約テスト（summary の形、xp 付与の検証、accept/decline/set の遷移、不正値 400）
- プレースメント評価: ClaudeRunner 注入のフェイクで rationale/stage の整形を検証（実プロンプトの品質はユーザー実走で確認）
- 既存テストは全部そのまま通ること（メニュー API の params 値変化はテスト側の期待値を更新）

## 12. スコープ外（明示）

- AE エラー密度・調音速度などの高度な昇格シグナル（M4 のメトリクス基盤で精緻化）
- ストリーク・バッジ・リーダーボード（研究制約により恒久的に対象外）
- 暗記例文300 の SRS 間隔のレベル連動（現行の固定ラダーを維持）
- プレースメントの発音評価（明瞭性評価は M4 以降）
