## 2. 構造的な問題（個別バグではなく設計の穴）

1. **会話系（自由会話・ロールプレイ）が支援ゼロ地帯**。AI は「Never switch to Japanese」「訂正しない」を指示されており、発話の意味を確認する手段も、詰まったときに言い方を尋ねる手段もない。一番負荷の高いメニューに一番支援がない逆転が起きている。
2. **「表示制御」を「データ削除」で実装している**。stage4+ ではサーバがチャンクの ja を空文字にして返すため（coach.ts の hintLang="en" 処理）、上級者が「今回だけ訳を見たい」と思っても復元不可能。表示の既定は stage で変えてよいが、データは常に持ち、見せるかはクライアント側で決めるべき。
3. **ユーザー調整の唯一の手段が「レベル一括変更」で粒度が粗い**。「秒数は今のままで日本語ヒントだけ欲しい」ができない。hideNote（例文300のみ）以外に個別トグルが存在しない。
4. **同じ機能が画面によってあったりなかったりする非対称**。訳解説ボタン: シャドーイングにあってライブラリにない。「もっと詳しく」: 例文にあってチャンクにない。cloze: 例文系にあって音読ウォームアップにない。
5. **i18n が新しい画面だけ**。練習系画面（ウォームアップ/4/3/2/振り返り/ロールプレイ/自由会話本体/シャドーイング/ライブラリ/ChunkList）は日本語ハードコード。加えてサーバ生成のメニュータイトル（「音読ウォームアップ」等）は EN UI でも日本語で配信される。

## 3. 次期計画（優先度順）

### P1: 会話系の日本語支援（最大ギャップの解消）
- 自由会話・ロールプレイの各 AI 発話バブルに **「訳」ボタン**（既存 `/api/coach/talk-explain` の軽量版: 訳のみ・ハッシュキャッシュ流用）
- **「言い方ヒント」ボタン**: 詰まったとき、言いたいことを日本語で入力（またはそのまま録音）→ 使える英語表現を2〜3個提案（新エンドポイント・会話コンテキスト付き）。研究上も「その場で必要になった表現」は定着が良い（need-based learning）
- ロールプレイに軽量準備: シナリオの開始フレーズ2〜3個（生成 or scenario md に追記）を開始前に表示

### P2: 学習サポート設定の統一（「おまかせ/多め/少なめ」）
- サイドバーに **サポート設定**（1つのセグメント: おまかせ=stage駆動 / 多め / 少なめ）＋個別トグル（日本語ヒント / モデルトーク / cloze既定）
- サーバ修正: チャンク ja を**常にデータとして返し**、表示既定のみ stage で決める（上記 §2-2 の設計是正）。モデルトーク mode=none 廃止 → button に統一（stage6 でも聞く手段は残す）
- 設定は localStorage（クライアント責務）。サーバの stage 駆動は「既定値の供給者」に格下げ

### P3: 段階的ステップの横展開
- シャドーイング: **「スクリプトを隠して聞く」を既定**にし、「スクリプトを表示」ボタンで開く（現状の逆）。訳解説は表示後に出す
- 音読ウォームアップ: 2周目として「歯抜けで音読」ステップ（既存 clozeText 流用・任意）
- 振り返り・AE の fixes に「もっと詳しく」ボタン（既存 explain 基盤流用）

### P4: 練習系画面の i18n 一括対応
- Warmup / 4/3/2 / Reflection / Roleplay / FreeTalk本体 / Shadowing / Library / ChunkList を STR 辞書化（機械的変換・辞書は既存構造に追記）
- サーバ生成のメニュータイトルは kind+topic を返してクライアントで組み立てる方式に変更（HTTP additive: title は残し titleKey を追加）

### P5: 小粒の非対称解消
- ライブラリに訳解説ボタン（fetchTalkExplanation 流用・1行）
- チャンクカードに「もっと詳しく」（プロンプトはチャンク用に調整）
- placement お題本文の日本語訳トグル（低ステージ受験者向け）
- cloze マスク率の設定（P2 の個別トグルに含めるか検討）

## 4. 実施メモ

- P1 と P2 は独立して着手可能。P3 は P2 の設定基盤があると綺麗（シャドーイングの既定値をサポート設定に従わせる）
- P4 は機械的だが量が多い（8画面）。1タスクで一括より 2 分割が安全
- 研究制約（情報的フィードバックのみ）は全案で維持。P1 の「言い方ヒント」は明示的な要求ベースなので recast 論争とは無関係
- 監査の生所見（file:line 付き 44件）はワークフロー出力に保存済み。実装計画を書く際は該当画面の所見を貼り付けて使う
