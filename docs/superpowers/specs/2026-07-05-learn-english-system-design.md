## 1. 背景と目的

- okash1n 専用の英会話学習システム。毎日30〜60分、この Mac 上で完結して回せること。
- English Boot Camp（2日間・アウトプット集中型）で効果を実感した。その路線（手持ちの英語で伝え切る訓練＝セカンドベストイングリッシュ、タスク型アウトプット練習）を、研究ベースで毎日の習慣に分解する。
- 本人特性: 日本語でも雑談は苦手。目的のある会話（仕事・技術の話）は得意。ghost-writer リポジトリの記事群（ID管理・セキュリティ・AIガバナンス等）が「すらすら話せるドメイン」。
- ただし本人特性に過剰適合させず、**方法論は最新の第二言語習得（SLA）研究を根拠にする**（ディープリサーチ実施中、結果は §5 に反映）。

## 2. 確定した前提（ヒアリング結果）

| 項目 | 決定内容 |
| --- | --- |
| ゴール場面 | (1) 仕事の実務会話（会議・技術討議・ベンダー対応） (2) 技術発信・登壇（記事内容を英語で語る、カンファレンスQ&A） |

| 時間 | 毎日30〜60分 |
| 期限感 | 3ヶ月で目に見える変化（チェックポイント例:「30分の英語技術ミーティングをAI相手に完走」）。毎月スピーキング測定 |
| 月額コスト | 追加数千円まで。音声認識はローカル無料、音声合成のみAPI、対話AIは既存の Claude サブスク |

## 3. ゴールと成功基準

- **3ヶ月チェックポイント**: 30分の英語技術ミーティング相当のセッションを、AI相手に途中で崩壊せず完走できる。
- **毎月測定**: スピーキング測定タスク（同一形式）で、発話速度・ポーズ・発話量などの流暢性指標が改善傾向にある。
- **継続**: 週5日以上の実施が3ヶ月続いている（継続自体を成功基準に含める）。

## 4. システム構成

### 4.1 技術スタック

| 層 | 採用 | 理由・備考 |
| --- | --- | --- |
| アプリ本体 | Bun + TypeScript サーバ / React + Vite のブラウザUI | localhost 専用・シングルユーザー。このリポジトリ内に置く |
| 音声認識（自分の声→テキスト） | whisper.cpp（ローカル、large-v3-turbo、英語専用モード） | 無料・音声が外に出ない。M5 Pro で1発話1〜2秒。brew で導入（セットアップスクリプト化） |
| 音声合成（AIの声） | OpenAI TTS API | 自然な英語音声で毎日のリスニング品質を確保。想定 月数百〜2千円。インターフェースを抽象化し、障害・オフライン時は macOS `say` にフォールバック。将来ローカルTTSに差し替え可 |
| 対話・コーチAI | Claude Agent SDK（Max サブスク経由、追加費用なし） | レッスン種別ごとのシステムプロンプトで「会話相手役」「コーチ役」を演じ分け |
| データ | リポジトリ内のプレーンファイル（Markdown / JSON / JSONL） | ~~リポジトリは private。テキストは全部コミット~~ **2026-07-05 改訂（OSS公開に伴う変更）**: 学習データ（`data/` 以下すべて — セッションログ・進捗・録音・キャッシュ）はコミットせずローカル専用（`data/.gitignore` で全無視）。「Claude Code が後から分析に使える」要件はローカルファイル読み取りで満たす |

> **2026-07-17 改訂注記（音声合成の行）**: 「障害・オフライン時は macOS `say` にフォールバック」は v0.29.1 で改訂済み。現行の TTS は macOS `say` / OpenAI 公式 / OpenAI 互換の**明示3値選択**（既定 `say`）で、明示選択したプロバイダの失敗時に別エンジンへ自動フォールバックしない（`app/server/tts.ts`・`app/server/tts-provider-store.ts` 参照）。

### 4.2 コンポーネント分割（各々独立にテスト可能）

1. **voice-pipeline** — 音声認識・音声合成のラッパ（`/stt`, `/tts` エンドポイント）
2. **coach** — Claude セッション管理。タスク種別プロンプト、会話相手役、セッション後フィードバック生成
3. **curriculum** — 「今日の30〜60分メニュー」を方法論ルール＋復習期日＋進捗データから自動構成
4. **srs** — 定型表現（チャンク）・語彙の間隔反復スケジューラ（産出リトリーバル前提。§5.4 参照）
5. **metrics** — 発話時間・語数/分・ポーズ等の自動計測、月次アセスメント
6. **web UI** — セッション実行画面（今日のメニュー・録音・再生・トランスクリプト）＋進捗ダッシュボード

### 4.3 会話1ターンのデータフロー

```
ブラウザで録音 → /stt (whisper.cpp) → テキスト
→ coach (Claude) が応答テキスト＋コーチ用内部メモを生成
→ /tts (OpenAI TTS) → ブラウザで音声再生
```

- ターン間の待ちは 3〜8 秒想定。ターン制の練習としては実用範囲。
- 将来、より会話らしい体験が欲しくなったら Realtime API モードを追加できる構造にしておく（v1 の対象外）。

### 4.4 セッション外のコンテンツ生成

- ghost-writer の記事・実務シナリオから、ロールプレイ設定・4/3/2 トピック・チャンク候補を Claude Code で随時生成し `content/` に蓄積する。
- 高頻度3,000ワードファミリーのうち「産出できない語」を洗い出す診断タスクもここで生成する（§5.4 のソース4系統に対応）。
- 日々のセッション実行とは分離（セッション中に重い生成はしない）。

### 4.5 エラー処理

- TTS API 障害 → macOS `say` にフォールバックしてセッション継続。
- 音声認識失敗・無音 → 再録音UIで即やり直し。
- セッションログは JSONL 追記型。クラッシュしても記録が消えない。
- オフライン時 → 音声対話以外（独り練習系タスク）は完走可能。

> **2026-07-17 改訂注記**: 「TTS API 障害 → `say` フォールバック」は v0.29.1 で改訂済み（§4.1 の注記と同じ）。TTS は明示3値選択・既定 `say` で、明示選択時は自動フォールバックせず失敗をユーザーに提示する。

### 4.6 テスト方針

- curriculum / srs / metrics: ユニットテスト（決定的ロジックなので相性が良い）
- voice-pipeline: フィクスチャ音声ファイルでの結合テスト
- web UI: 最小のスモークテスト＋日常使用による手動確認

### 4.7 リポジトリ構成（想定）

```
learn-english/
  app/          # Webアプリ（サーバ + クライアント）
  curriculum/   # 方法論ドキュメント、タスク種別定義
  content/      # 生成した練習素材（ロールプレイ設定、描写課題、チャンク候補）
  data/         # セッションログ、進捗メトリクス、SRS状態（音声のみ gitignore）
  docs/         # 設計ドキュメント、リサーチレポート
  scripts/      # セットアップ（whisper導入・モデルDL・APIキー検証）
```

### 4.8 v1 の非対象（将来オプション）

- Realtime API による低遅延音声対話モード
- 発音の音素レベル解析・スコアリング
- Apple Foundation Models（macOS 26 の on-device AI）を使った即時ヒント
- iPhone / 外出先対応

## 5. 学習方法論と日々のメニュー

方法論はディープリサーチ2回（計49クレームを3票敵対的検証、48確認・1棄却）とスポット確認に基づく。各原則に検証レベルを明示する。

> **2026-07-17 訂正注記**: 上記の検証統計は集計誤り。正しくは**計50クレームを3票敵対的検証、49確認・1棄却**（第1弾: 25クレーム→25確認・0棄却、第2弾: 25クレーム→24確認・1棄却）。一次資料は[第1弾レポート](../../research/2026-07-05-sla-speaking-research-phase1.md)・[第2弾レポート](../../research/2026-07-05-sla-speaking-research-phase2.md)それぞれの冒頭統計。

### 5.1 方法論の原則（10箇条）

| # | 原則 | 検証レベル |
| --- | --- | --- |
| 1 | **時間の約75%を実際に話すことに使う**（Nation の会話コース処方: 意味重視アウトプット~50%＋流暢性活動25%） | 検証済み（枠組み。比率自体は専門家処方） |
| 2 | **流暢性開発は「100%既知の内容」×「時間圧」×「反復」**。自分の技術記事・仕事の話はこの条件を満たす理想素材（＝本人の直感は研究的に正当） | 検証済み・高 |
| 3 | **4/3/2 法を流暢性の主エンジンに**: 同じ話を4分→3分→2分、毎回「別の聞き手」設定。効果は新トピックにも転移（d=0.96–2.35） | 検証済み・高 |
| 4 | **正確さは「反復間の遅延明示フィードバック」で補完**: 発話中は流れを止めず、反復の合間の約1分でエラーログ＋説明を渡す（4/3/2+AE 方式）。時間圧単体では正確さは伸びない | 検証済み（単一研究のため中） |
| 5 | **同一タスクのセッション内反復は3回まで**、反復はセッション間に分散（日本人A2-B1の直接データ: 6回連続は逆効果） | 検証済み・中 |
| 6 | **チャンク（定型表現）と語彙は「産出リトリーバル＋分散」**: 文を作って言う形式が受容型演習より産出知識を作る。1項目3〜4回で逓減、多数項目を少数回ずつ。SRS間隔は単純な均等でよく（拡張間隔に実証優位なし）、長期保持には長めの間隔 | 検証済み・高〜中 |
| 7 | **語彙コアは高頻度〜3,000ワードファミリー＋実務定型表現**（「2,000語で十分」説は否定済み）。ITドメイン語彙はその上に積む。SRSは入口にすぎず、口頭使用への接続を必須にする（意図学習は遅延で産出側が25%まで減衰する） | 検証済み・高 |
| 8 | **シャドーイングは「知覚ドリル」として少量**（1日5〜10分、インプット枠）: A2-B1ではリスニング技法であり、スピーキング練習の代替にしない（自発発話への転移証拠は限定的） | 検証済み・高 |
| 9 | **発音は高機能負荷の対立に絞った少量の HVPT 型知覚ドリル**（週2〜3回×5分、複数TTSボイスで minimal pair）。知覚訓練は産出にも d≈0.54 転移。明瞭性優先、アクセント矯正はしない | スポット確認のみ |
| 10 | **習慣設計**: ユーザーが決める if-then アンカー（d≈0.65）、週5達成ベースの緩いストリーク（1日欠落を罰しない）、開始1クリック、90日を習慣化地平線（自動化の中央値66日をカバー） | スポット確認のみ |

> **2026-07-17 改訂注記（原則10）**: 「週5達成ベースの緩いストリーク」はストリーク自体を導入せず、練習日カレンダー＋練習記録（今週◯日・累計◯日の情報的表示のみ）へ方針置換済み。根拠は[第3弾リサーチ](../../research/2026-07-05-continuation-research-phase3.md)の統合設計推奨 D-4（失うと痛いストリークを避け、情報的フィードバックの最小構成にする）と AGENTS.md の binding 制約（喪失演出を導入しない）。
>
> **2026-07-17 改訂注記（原則9）**: 発音の HVPT 型知覚ドリルは現行アプリでは未導入（v0.29 時点。§4.8 の音素レベル解析と同様、将来オプション扱い）。
>
> **2026-07-17 訂正注記（原則10・効果量）**: if-then アンカーの効果量 **d≈0.65**（Gollwitzer & Sheeran 2006）は、原著者ら自身の2024年メタ分析（Sheeran, Listrom & Gollwitzer 2024・642検証・出版バイアス補正後）で **d=0.15** に下方修正済みであり、**d≈0.65 は使用禁止**（[第3弾リサーチ](../../research/2026-07-05-continuation-research-phase3.md) B-8〜10 と[スポット確認](../../research/2026-07-05-spot-checks.md)「実装意図」節の訂正参照）。効果は実在するが小さく、継続の主柱ではなく**補助策**の位置づけ。この項目の検証レベルは「スポット確認のみ」ではなく第3弾で3票検証済み（訂正反映）。

AI を会話相手にすること自体は、チャットボット学習効果のメタ分析（Hedges' g = 0.608、31研究）が中程度の効果を支持（検証済み・高）。ただし音声型LLM特有の効果・リスクの実証はまだ薄く、月次測定で「実際に効いているか」を自前で検証する（§5.5）。

### 5.2 デイリーメニュー（60分版）

| 順 | ブロック | 時間 | 内容 |
| --- | --- | --- | --- |
| 1 | **チャンク産出リトリーバル** | 8分 | 期日が来たチャンク・語彙を「文を作って口頭で言う」形式で復習（音声認識で確認）。新規チャンクの導入もここ |
| 2 | **流暢性: 4/3/2＋AE** | 16分 | 自分の記事・直近の仕事から選んだトピックで 4分→(AIのエラーログ＋説明 約1分)→3分→2分。AIは毎回「別の聞き手」を演じる |

| 4 | **インプット/知覚** | 8分 | coach が翌日の 4/3/2 トピックの「模範トーク」を生成し TTS 音声化したものをシャドーイング（知覚ドリル。翌日の自分の発話の下敷きにもなる）、または発音マイクロブロック（週2〜3回） |

### 5.4 チャンク・語彙パイプライン

```
ソース（4系統）                     学習フロー
1. セッション中に詰まった表現   →  新規導入（意味・例文・音声）
2. ghost-writer 記事からの抽出  →  産出リトリーバル（文を作って言う×3〜4回、
3. 会議・登壇の定型表現リスト       日をまたいで分散: 例 1→3→7→21→60日）
4. 高頻度3,000のうち産出不能語  →  4/3/2・ロールプレイ内で実使用を検出
                                →  卒業（保守間隔へ）
```

- SRS 実装は単純な固定間隔スケジュール（均等でよい）。口頭産出の成否で昇格・降格。
- 「実使用の検出」= セッショントランスクリプトから coach が学習中チャンクの使用を検出してマークする。

### 5.5 測定（毎セッション自動＋週次＋月次）

| 頻度 | 測定 | 指標 |
| --- | --- | --- |
| 毎セッション | 自動ログ | 発話時間・総語数・WPM・AI訂正数・詰まり検出数 |
| 週1回 | **2分測定モノローグ**（毎回新規プロンプト、形式固定） | **調音速度・節内ポーズ頻度・繰り返し頻度**（知覚熟達度の検証済み予測子）＋誤り数・語彙多様性 |
| 月1回 | **30分模擬ミーティング完走テスト** | 完走可否・主観負荷・coach の質的評価。3ヶ月チェックポイントの直接測定 |

- 節内ポーズは whisper の単語タイムスタンプ＋Claude の節境界アノテーションから推定（近似値であることをダッシュボードに明記）。
- 流暢性指標だけでは熟達度の6割しか説明できない（検証済み）ため、正確さ・語彙指標を必ず併記する。

> **2026-07-17 改訂注記**: 現行実装は単語タイムスタンプ＋節境界アノテーションを未導入で、週次のポーズ指標はセグメント間ギャップ（>300ms）合算の「ポーズ比率」による近似になっている（節内・節末を区別できない）。この限界と推定値である旨は進捗画面の各カードと [指標定義](../../metrics.md) に明記し、録音が疎な週は率・前週比を確定表示しない。月次レビューの生成指示にも「ポーズ全体を減らす助言をしない」制約を組み込み済み（Issue #183 / #217）。節内ポーズ頻度そのものへの置換は将来対応。

### 5.6 習慣化の仕組み（アプリ側の実装）

- 初回セットアップでユーザーが if-then アンカーを1つ決める（例:「朝コーヒーを淹れたら開く」）。アプリはそれを表示するだけ（通知スパムはしない）。
- ストリークは「週5日以上」を単位にカウント。1日欠けても壊れない。
- 起動 → 今日のメニューが既に生成済み → 1クリックで開始（開始摩擦の最小化）。
- ダッシュボードに週次測定のトレンドを常時表示（進捗の可視化を動機づけに使う）。

> **2026-07-17 改訂注記**: 「ストリークは『週5日以上』を単位にカウント」はストリーク自体を導入せず、練習日カレンダー＋練習記録（今週◯日・累計◯日の情報的表示のみ）へ方針置換済み（§5.1 原則10 の注記と同じ。[第3弾リサーチ](../../research/2026-07-05-continuation-research-phase3.md) D-4・AGENTS.md の binding 制約参照）。

## 7. リスクと対応

| リスク | 対応 |
| --- | --- |
| ターン遅延（3〜8秒）が練習の没入感を削ぐ | M1 で実測し、許容できなければ応答の先読み生成・ストリーミングTTSで短縮。将来は Realtime API モード |
| AI が寛容すぎて誤りが放置される（研究でも未検証のリスク） | coach プロンプトで「AE 方式の明示フィードバック」を必須化し、訂正数をログで監視 |
| 節内ポーズ推定の精度が低い | 近似と明記し、同一手法での相対トレンドのみを見る（絶対値は使わない） |
| 中だるみ（4〜8週目） | 週5ストリーク・週次トレンド表示・月次模擬ミーティングで中間の手応えを作る。8週目に方法論レビューを予定 |
| 方法論の未検証部分（発音・習慣化・音声型LLM効果） | 「小さく・可逆に」実装し、月次測定で効果を自前検証。将来のリサーチ第3弾で更新可能 |

> **2026-07-17 改訂注記（中だるみ行）**: 「週5ストリーク」はストリーク自体を導入せず、練習日カレンダー（情報的表示）へ方針置換済み（§5.1 原則10 の注記参照。[第3弾リサーチ](../../research/2026-07-05-continuation-research-phase3.md) D-4・AGENTS.md の binding 制約）。
